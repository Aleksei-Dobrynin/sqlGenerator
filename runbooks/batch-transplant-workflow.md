# Batch-transplant workflow (предпочтительный режим работы агента)

> **Что это.** Каноничный процесс, которым AI-агент генерирует код утилитой `sql-generator` и
> встраивает результат в целевой проект. Рассчитан на агента и опирается на **компилятор/тулчейн как
> детерминированный оракул ошибок**: один прогон диагностики вместо N пер-сущностных сборок, массовые
> скриптовые правки вместо пофайловых.
>
> Это **рекомендуемый по умолчанию** способ для full-module / multi-table scaffolding. Для одиночной
> простой таблицы достаточно `QuickGenerate` (см. [`agent-spec.md`](../agent-spec.md) → Workflows).

## Два базовых принципа

1. **Утилита = MCP-сервер.** Агент вызывает её как MCP (`save_schema`, `generate_files`,
   `list_presets`, …), а не запускает CLI вручную. Инструменты принимают **только пути к файлам**, не
   содержимое (token economy — см. [`agent-spec.md`](../agent-spec.md)).
2. **Агент парсит схему сам.** Для любого нетривиального DDL (боевой pg_dump, quoted-идентификаторы,
   составные ключи, partitioning и т.п.) агент парсит SQL **самостоятельно** по промпту
   `sql_parsing_instructions` и подаёт результат через `save_schema`. Встроенные парсеры
   (`parse_sql`/regex и `--use-llm`) — для простого DDL и не-агентного CLI; в этом воркфлоу они **не на
   критическом пути**. Причина: regex-парсер исторически спотыкается на боевых дампах (quoted-имена,
   table-level/ALTER PK), а агент разбирает схему точнее и сразу в нужный JSON.

## Воркфлоу: 5 фаз

```
[предусловие: инфра-базис заведён в проекте]
        │
   SQL DDL
        ▼
  1. GENERATE (batch)      агент парсит сам → save_schema → generate_files
        ▼                  вся партия в scratch (не в проект)
  2. BULK TRANSPLANT       одна скриптовая операция: папка_генератора → слой
        ▼
  3. DIAGNOSTICS ORACLE    один прогон build + typecheck/build + lint → ПОЛНЫЙ список ошибок
        ▼
  4. BATCH-FIX             чинить классами, каскадные → частные; build до зелёного ⟲
        ▼
  5. VERIFY                CRUD / runtime к живой базе
```

### Предусловие — инфра-базис

Инфраструктурные типы, которые ожидает сгенерированный код (напр. `BaseLogDomain`,
`RepositoryException`, `PaginatedList`, `IUnitOfWork`, `*Helper`, `Infrastructure.Data.Models`),
должны существовать в проекте **до** фазы 3. Иначе список ошибок диагностики тонет в каскадных
«type not found». Конкретный набор типов **зависит от проекта и пресета** — определите его по эталону
(существующий модуль того же пресета) перед стартом.

### Фаза 1 — Generate (batch)

1. `list_presets(templatesDir?)` — выбрать пресет (`default`, `clean-arch`, …).
2. **Парсинг.** Получить промпт `sql_parsing_instructions(sql?)`, распарсить SQL в schema JSON
   (формат — см. [`agent-spec.md`](../agent-spec.md) → Schema JSON Format). Записать JSON на диск.
   - Для **простого** DDL допустимо `parse_sql(sqlFilePath)` (regex), но проверьте результат.
3. `save_schema(schemaFilePath, outputPath)` — валидация. Критические ошибки (пустой `TableName`,
   отсутствие PK, дубли колонок, FK на несуществующую колонку, невалидный C#-тип) → `success: false`;
   некритичные → `warnings[]`. Исправить и повторить до `success: true`.
4. `generate_files(schemaFile, outputDir, templatesDir?, presetName?, includeVirtualFks?)` — вся партия
   генерируется в **scratch** (напр. `db/_codegen/out`), **не** в дерево проекта.

### Фаза 2 — Bulk transplant

Перенести все файлы в слои проекта **одной скриптовой операцией** по фиксированному маппингу
`папка_генератора → слой`. Маппинг **настраивается под архитектуру проекта**; типовой пример для
layered/`default`:

| Папка генератора | Слой проекта (пример) |
|------------------|-----------------------|
| `entity` | `Domain` |
| `irepo`, `usecase`, `dto` | `Application` |
| `repo` | `Infrastructure` |
| `controller` | `WebApi` |
| frontend `*View`, `integration`, `api`, `constants` | `frontend/src` |

Без пошагового редактирования по файлу — это массовое перемещение.

### Фаза 3 — Diagnostics oracle

Один прогон диагностики, собрать **полный** список ошибок (а не первую):

```bash
dotnet build              # backend
npm run typecheck         # frontend (или npm run build)
# + линтер проекта
```

Детали и почему «полный список, а не первая ошибка» — [`docs/batch/diagnostics-oracle.md`](../docs/batch/diagnostics-oracle.md).

### Фаза 4 — Batch-fix по классам ошибок

Сгруппировать ошибки по типу и чинить **класс целиком** массовой/скриптовой правкой, в порядке
**каскадные → частные**:

1. недостающие инфра-типы;
2. namespace / `using`;
3. регистрация DI;
4. повторяющиеся баги шаблонов (документировать — см. ниже).

Цикл build повторяется до зелёного. Таксономия классов и порядок — [`docs/batch/error-classification.md`](../docs/batch/error-classification.md).

### Фаза 5 — Verify

Проверить CRUD/runtime к живой базе. Чек-лист — [`docs/batch/crud-runtime-testing.md`](../docs/batch/crud-runtime-testing.md).

## Дефекты шаблонов — политика

Найденные баги шаблонов генератора **документируются для ревью**, а не правятся «на лету» во время
переноса. Повторяющийся дефект можно временно обойти batch-постобработкой на стороне проекта (фаза 4),
но исправление самого шаблона — отдельная задача в репозитории утилиты. Это разделяет контур
«потребитель чинит свой проект» и контур «мейнтейнер чинит утилиту».

## Источник

Воркфлоу обобщён из реального применения (проект-потребитель `bga-cabinet`, ADR-0002
«Batch-воркфлоу переноса сгенерированного кода, compiler-as-oracle»). Проектно-специфичные детали
(конкретные namespace, набор инфра-типов) в этом документе приведены как **пример** и настраиваются
под целевой проект.
