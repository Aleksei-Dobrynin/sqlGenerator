# Отчёт: версионирование пресетов ↔ привязка к проектам

> Памятка по сессии 2026-06-09. Фиксирует **принятое** и **открытое** по вопросу:
> как вести синхронизированную историю версий шаблонов (пресетов) генератора с
> проектами-потребителями, не храня scaffold'ы проектов в репо генератора.
> Документ самодостаточен — его можно показать агенту внутри проекта-потребителя,
> чтобы проработать механизм биндинга на уровне проекта (см. § «Открытая тема: формат биндинга»).

**Статус:** дизайн выбран (Подход 1); детали реализации (OD-1…OD-6) и формат биндинга (§7.1…§7.7) — **залочены** (сессия 2026-06-11). Готово к spec + плану реализации.

---

## 1. Проблема

- Утилита генерит рабочий бойлерплейт-код под конкретную **архитектуру**. Архитектура воплощена в **пресете** (`templates/<preset>`).
- Пресеты — это **параллельные семейства архитектур**, не версии друг друга: `default` — одна архитектура, `clean-arch` — другая. Несколько проектов работают на `default`, несколько на `clean-arch`.
- **Боль:** внутри одного пресета архитектура со временем эволюционирует. Проекты на одном пресете собирались в разные моменты. Поправив пресет под новую работу, теряем возможность догенерить код (новую таблицу/модуль) под **старый prod-проект** на том же пресете — новый шаблон даёт уже другой код, а кардинально рефакторить вышедшее в прод приложение не хочется.
- Изменения внутри пресета **вперемешку** (аддитивные и ломающие, заранее непредсказуемо) → нельзя полагаться на «последний пресет всегда ляжет в старый проект». Нужна **настоящая привязка проекта к конкретной версии пресета**.
- **Ограничение:** не раздувать репо генератора хранением scaffold'ов всех возможных проектов.

## 2. Ключевой вывод

Scaffold'ы проектов хранить в репо генератора **не нужно вообще**. Достаточно трёх вещей:

1. **История версий каждого пресета** — это уже есть, git-история репо генератора.
2. **Привязка «проект → (пресет, версия)»** — живёт в репо проекта, не в генераторе.
3. **Способ извлечь шаблоны нужной версии** в момент генерации.

«Синхронизированная история» = git-история проекта (где менялась привязка) ↔ git-теги генератора, связанные через `ref`. Никакого дублирования.

## 3. ПРИНЯТО: Подход 1 — git-native (выбран пользователем)

```
Репо генератора (как есть)          Репо проекта P (как есть)
├─ templates/default/      ←──┐     ├─ .sqlgen.json  { preset, ref }  ← привязка
├─ templates/clean-arch/      │     └─ src/...                          (git-tracked)
└─ git-теги: default-v1.0 ────┘
            clean-arch-v1.0
   (= именованные указатели на коммит,
    где папка пресета имела версию X)
```

- **Версии = git-теги** на репо генератора. Тег `default-v1.1` — имя для «коммита, где `templates/default` был версии 1.1». Инструмент извлекает **только папку нужного пресета** из тега, поэтому неважно, что ещё в том коммите.
- **Привязка** — крошечный `.sqlgen.json` в корне репо проекта. Репо генератора о проектах не знает → не раздувается.

### Сравнение, на основании которого выбран Подход 1

| | Где версии | Привязка | Раздувание репо генератора | Минусы |
|---|---|---|---|---|
| **1. Git-native (ВЫБРАН)** | git-теги генератора | `.sqlgen.json` в проекте | **нулевое** | нужен плумбинг чтения шаблонов на ref; дисциплина тегов |
| 2. Версионированные папки `templates/default/v1`, `/v2` | копии деревьев | `preset: default@v2` | частичное (копии пресетов) | дублирование; аддитивные правки ломают воспроизводимость без форка |
| 3. Пресеты как отдельные пакеты/submodule с semver | отдельные репо | как зависимость | нулевое | тяжёлая инфра, оверкилл для 2 архитектур |

## 4. Дизайн Подхода 1 (детали — со статусом «открыто», см. § 6)

### 4.1 Модель версионирования (per-preset теги)
Каждый пресет версионируется независимо: `default-v1.0`, `clean-arch-v1.0`, далее `default-v1.1`, `default-v2.0`…
Схема имени: `<preset>-v<major>.<minor>`. Ломающее изменение → bump **major**; аддитивное → bump **minor**.
Опционально in-tree маркер `templates/<preset>/preset.json` = `{ "name": "default", "version": "1.0" }` (провенанс в дереве, `list_presets` сможет отдавать версию).

### 4.2 Привязка в проекте — `.sqlgen.json` (корень репо проекта, под git)
```json
{
  "preset": "default",
  "generatorRef": "default-v1.0",
  "generatorRepo": "https://github.com/Aleksei-Dobrynin/sqlGenerator.git"
}
```
История этого файла в проекте = «когда и на какую версию архитектуры перешли».

### 4.3 Поток генерации с резолвом версии
```
1. Инструмент читает .sqlgen.json проекта → { preset: default, ref: default-v1.0 }
2. В репо генератора: git worktree add --detach <temp> default-v1.0
3. Генерация с templatesDir = <temp>/templates/default
4. git worktree remove <temp>
```
`git worktree` — извлекает дерево на теге в отдельную папку, не трогая рабочий чекаут; на Windows штатно.
Альтернативы: `git archive <tag> | tar -x` (зависит от tar), `git show <tag>:path` (пофайлово).

### 4.4 Изменения в CLI / MCP (минимальные, переиспользуют `templatesDir`)
- **CLI:** флаг `--templates-ref <ref>` — материализует `templates/` на ref в temp и подставляет как templates-dir.
  Эргономика: `--from-project <path>` — читает `.sqlgen.json` проекта, сам выводит `preset` + `templates-ref`.
- **MCP:** добавить `templatesRef?` в `generate_files` и `quick_generate` (экономия токенов не нарушается — это строка-ref, не содержимое).

### 4.5 Шаги «оформить текущее состояние как v1.0»
1. (опц.) Добавить `preset.json` с `version: "1.0"` в каждый пресет, закоммитить.
2. Теги на текущий HEAD: `git tag default-v1.0 && git tag clean-arch-v1.0`, запушить.
3. В каждом существующем проекте добавить `.sqlgen.json` с его пресетом и `ref: <preset>-v1.0`, закоммитить **в репо проекта**.

### 4.6 Известное ограничение
Подход пинит **шаблоны**, не бинарь генератора (запускается текущий парсер/генератор). Обычно архитектуру несут шаблоны → для цели хватает. `generatorRef` указывает на полный коммит репо, так что при критичном расхождении логики парсера можно собрать и запустить генератор на том же ref (тяжелее). Старт: пиним только шаблоны.

## 5. ПРИНЯТЫЕ решения (зафиксировано в этой сессии)

- ✅ **A-1.** Модель связи: 1 пресет → N проектов; пресеты — параллельные независимые архитектуры, не версии.
- ✅ **A-2.** Характер изменений внутри пресета — вперемешку (аддитивные + ломающие), непредсказуемо → нужна жёсткая привязка проекта к версии.
- ✅ **A-3.** Выбран **Подход 1 (git-native)**: версии = git-теги генератора, привязка = манифест в репо проекта, scaffold'ы не хранятся.
- ✅ **A-4.** Текущее состояние пресетов оформляется как **v1.0** (`default-v1.0`, `clean-arch-v1.0`).

## 7. Открытая тема: формат биндинга (прорабатывается с агентом В ПРОЕКТЕ)

Пользователь хочет **отдельно продумать формат биндинга конкретной версии шаблона с конкретным проектом** — на уровне проекта-потребителя, показав этот отчёт агенту внутри проекта. Вопросы к проработке (вход для project-агента):

- **Где и в каком формате** проект объявляет привязку (`.sqlgen.json` — кандидат из OD-3; альтернативы?).
- **Гранулярность привязки:** один ref на весь проект, или возможны разные версии пресета для разных модулей/сущностей внутри одного проекта?
- **Как проект инициирует догенерацию** старого модуля: вызывает генератор как CLI / через MCP / через скрипт-обёртку в репо проекта.
- **Маппинг «папка генератора → слой проекта»** при переносе (связано с batch-transplant воркфлоу, см. § «Связанное»).
- **Обновление привязки** при сознательном переходе проекта на новую версию пресета (что считается «миграцией», как фиксируется в истории проекта).
- **Контракт между генератором и проектом:** генератор отдаёт версию через что-то машиночитаемое (тег + `preset.json`), проект это потребляет — согласовать обе стороны.

Результат этой проработки (формат биндинга) → вернётся сюда либо в новый отчёт/ADR, после чего можно лочить OD-3 и идти к реализации.
