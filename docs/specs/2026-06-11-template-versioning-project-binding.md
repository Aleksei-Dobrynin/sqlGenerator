# Spec: версионирование пресетов и привязка версии шаблона к проекту

> **Статус:** design locked (2026-06-11). Это **design-спека** — фиксирует *что* строим (форматы,
> контракты, поведение). Обоснование решений и trade-offs (*почему*) — в decision-log'е
> `template-versioning-report.md` (корень репо). Пошаговый *как* — в плане реализации (см. §10).

## 1. Цель и охват

Дать проекту-потребителю возможность **догенерировать код под закреплённую версию архитектуры**
(пресета), даже после того как пресет в репо генератора эволюционировал. Без хранения scaffold'ов
проектов в репо генератора.

**В охвате v1.0:** версионирование пресетов git-тегами, манифест привязки в проекте, резолвер
извлечения шаблонов на ref, поверхность CLI/MCP, оформление текущего состояния как v1.0.

**Вне охвата v1.0 (backlog, §9):** пиннинг бинаря генератора, tool-assisted diff версий, per-module
провенанс-маркер, per-module гранулярность привязки.

## 2. Модель версионирования

- Версия пресета = **git-тег** репо генератора. Схема имени: **`<preset>-v<major>.<minor>`**
  (`default-v1.0`, `clean-arch-v1.0`, далее `default-v1.1`, `default-v2.0`).
- **Гранулярность — per-preset** (OD-1): каждый пресет тегируется независимо. Bump одного пресета не
  затрагивает версии других.
- Bump **major** — ломающее изменение пресета; **minor** — аддитивное.
- Тег — авторитетный источник версии (резолвимый `ref`). In-tree файл (§4) зеркалит версию для
  провенанса и `list_presets`.

## 3. Манифест привязки — `.sqlgen.json`

Расположение: **корень репо проекта-потребителя**, под git. Формат — JSON. Одна привязка на весь
проект (OD §7.2: один `ref` на проект, не per-module).

```json
{
  "preset": "default",
  "generatorRef": "default-v1.0",
  "generatorRepo": "https://github.com/Aleksei-Dobrynin/sqlGenerator.git",
  "layerMap": {
    "entity": "Domain",
    "irepo": "Application",
    "usecase": "Application",
    "dto": "Application",
    "repo": "Infrastructure",
    "controller": "WebApi"
  }
}
```

| Поле | Обяз. | Назначение |
|------|-------|-----------|
| `preset` | да | имя пресета (`default`, `clean-arch`, …) |
| `generatorRef` | да | git-тег версии (`<preset>-v<major>.<minor>`) — `ref` для извлечения шаблонов |
| `generatorRepo` | нет | URL/путь репо генератора (если проект не держит генератор локально) |
| `layerMap` | нет | маппинг `папка_генератора → путь_в_проекте` для transplant. Нет → агент выводит маппинг как в batch-transplant фазе 2 |

История этого файла в git проекта = аудируемый лог миграций версии (§6).

## 4. In-tree маркер версии — `preset.json`

В каждом пресете: **`templates/<preset>/preset.json`**.

```json
{ "name": "default", "version": "1.0" }
```

- Зеркалит версию тега. **Синхрон тег ↔ `preset.json` — пункт чек-листа релиза версии** (§7).
- Даёт: `list_presets` репортит версию; самоописательность любого чекаута.
- Контракт (§7.6): тег — авторитет, `preset.json` — зеркало.

## 5. Резолвер и поток генерации

Единый модуль резолва (OD §7.3, общий для CLI и MCP):

```
1. Прочитать .sqlgen.json проекта → { preset, generatorRef }
2. В репо генератора: git worktree add --detach <temp> <generatorRef>      (OD-2)
3. templatesDir := <temp>/templates/<preset>
4. Сгенерировать с этим templatesDir (существующий пайплайн генерации)
5. git worktree remove <temp>
```

- Извлечение — **`git worktree add --detach`** (OD-2): Windows-native, без внешних зависимостей,
  атомарно. Тянется всё дерево тега; используется только подпапка пресета.
- Воспроизводимость v1.0 — **только шаблоны** (OD-6): запускается текущий бинарь генератора;
  `generatorRef` указывает на полный коммит, путь к сборке генератора на ref открыт на будущее.

## 6. Протокол миграции версии (§7.5)

Переход проекта на новую версию пресета = **осознанная миграция всего проекта** (не «новые модули на
v2, старые на v1» — гранулярность одна на проект):

1. Сменить `generatorRef` в `.sqlgen.json` на новый тег.
2. Закоммитить в репо проекта. **Git-история файла = лог «когда и на какую версию перешли».**
3. Реконсиляция существующего кода под новую архитектуру — **ответственность оператора**, вне контура
   механизма привязки.

## 7. Поверхность CLI / MCP (OD-5)

**CLI:**
- `--from-project <path>` — читает `.sqlgen.json` проекта, выводит `preset` + `generatorRef`, резолвит
  шаблоны (основной эргономичный путь).
- `--templates-ref <ref>` — низкоуровневый: материализовать `templates/` на ref напрямую (без
  манифеста).

**MCP** (`generate_files`, `quick_generate`):
- `templatesRef?: string` — ref для извлечения шаблонов.
- `fromProject?: string` — путь к проекту с `.sqlgen.json` (MCP сам резолвит preset+ref).

Token economy не нарушается: параметры — строки-пути/ref, не содержимое (ADR-0003).

## 8. Оформление текущего состояния как v1.0 (rollout)

1. Добавить `preset.json` (`version: "1.0"`) в каждый пресет, закоммитить.
2. Проставить теги на HEAD: `git tag default-v1.0`, `git tag clean-arch-v1.0`, запушить.
3. В каждом существующем проекте-потребителе завести `.sqlgen.json` с его пресетом и
   `generatorRef: <preset>-v1.0`, закоммитить **в репо проекта**.

## 9. Backlog (вне v1.0)

- Tool-assisted diff шаблонов между версиями (помощь в реконсиляции при миграции) — §7.5 B.
- Per-module провенанс-маркер `.sqlgen-provenance.json` в выводе — §7.7 A.
- Сборка и запуск генератора на ref (полная воспроизводимость логики) — OD-6 B.
- Per-module гранулярность привязки — §7.2 B/C.
