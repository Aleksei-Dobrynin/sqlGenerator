# Фаза 4 — Batch-fix: классы ошибок и порядок

> Часть [batch-transplant workflow](../../runbooks/batch-transplant-workflow.md). Здесь — таксономия
> классов ошибок и **порядок** их устранения.

## Принцип

Ошибки из фазы 3 группируются по типу и чинятся **классом целиком** массовой/скриптовой правкой —
не пофайлово. Порядок — **от каскадных к частным**: каскадные классы при исправлении схлопывают
множество зависимых ошибок, поэтому идут первыми.

## Порядок классов

| # | Класс | Типичный симптом | Как чинить (класс целиком) |
|---|-------|------------------|-----------------------------|
| 1 | **Недостающие инфра-типы** | `type or namespace 'BaseLogDomain' not found`, `RepositoryException`, `IUnitOfWork`, `*Helper` | Завести/подключить инфра-базис (предусловие). Каскадно убирает большинство «not found». |
| 2 | **Namespace / using** | `are you missing a using directive`, неверный namespace файла | Массово выровнять namespace/using по конвенции проекта (скриптом по папкам). |
| 3 | **Регистрация DI** | рантайм/композиционные ошибки резолва сервисов/репозиториев | Добавить регистрации классом (все новые репозитории/сервисы/usecase разом). |
| 4 | **Повторяющиеся баги шаблонов** | один и тот же дефект во **всех** сгенерированных файлах одного типа | Временный обход batch-постобработкой на стороне проекта. Сам дефект шаблона — **документировать для ревью** (см. ниже), не править на лету. |

После каждого класса — повторный прогон диагностики (фаза 3) до зелёного.

## Почему именно такой порядок

Каскадный класс (инфра-типы) генерирует десятки производных ошибок в namespace/DI/использованиях.
Починив его первым, вы убираете эти производные до того, как начнёте разбирать более частные классы —
иначе список раздут «ложными» ошибками-следствиями.

## Баги шаблонов — разделение контуров

- **Контур проекта (здесь):** временно обойти повторяющийся дефект массовой постобработкой, чтобы
  довести build до зелёного.
- **Контур утилиты:** зафиксировать дефект для ревью владельца утилиты; исправление шаблона —
  отдельная задача в репозитории `sql-generator`. Не править генератор «на лету» во время переноса.
