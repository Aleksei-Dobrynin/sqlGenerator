# Фаза 3 — Diagnostics oracle

> Часть [batch-transplant workflow](../../runbooks/batch-transplant-workflow.md). Здесь — детали третьей
> фазы: как использовать компилятор/тулчейн как **детерминированный оракул ошибок**.

## Идея

После bulk-transplant (фаза 2) не угадывайте, что сломалось. Запустите тулчейн **один раз** и получите
**полный** список ошибок — это вход для batch-fix (фаза 4). Компилятор знает точно: какие типы не
найдены, где не сходятся namespace, что не зарегистрировано в DI.

## Команды (пример)

```bash
# Backend (.NET)
dotnet build                       # вся солюшн целиком, не по проекту

# Frontend
npm run typecheck                  # или: npm run build
npm run lint                       # линтер проекта, если есть
```

Команды **зависят от проекта** — используйте те, что приняты в целевом репозитории.

## Правила

1. **Полный список, а не первая ошибка.** Не чините по одной сверху. Соберите весь вывод build/typecheck/
   lint целиком — частые ошибки повторяются десятками и чинятся одним классом (фаза 4).
2. **Один прогон на цикл.** Прогнали → собрали список → batch-fix класса → снова прогнали. Не пересобирайте
   после каждой отдельной правки внутри класса.
3. **Инфра-базис — заранее.** Если инфраструктурные типы не заведены (предусловие воркфлоу), список будет
   зашумлён каскадными «type/namespace not found». Сначала предусловие, потом диагностика.
4. **Сохраняйте сырой вывод.** Записать полный лог ошибок на диск (а не держать в контексте) — удобно
   группировать и сверять прогресс между циклами.

## Выход фазы

Структурированный список ошибок, сгруппированный по классам (см.
[`error-classification.md`](error-classification.md)) — вход для фазы 4 (batch-fix).
