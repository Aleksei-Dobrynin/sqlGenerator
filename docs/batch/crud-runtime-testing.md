# Фаза 5 — Verify (CRUD / runtime)

> Часть [batch-transplant workflow](../../runbooks/batch-transplant-workflow.md). Заключительная фаза:
> убедиться, что встроенный код **работает в рантайме**, а не только компилируется.

## Зачем

Зелёный build (фазы 3–4) подтверждает компиляцию, но не корректность SQL, маппинга и DI в рантайме.
Сгенерированные репозитории содержат реальный SQL (`INSERT`/`UPDATE`/`SELECT`) — его надо проверить
против живой базы.

## Чек-лист

1. **Поднять приложение** к живой/тестовой базе (та же схема, что парсилась).
2. **CRUD по 1–2 представительным сущностям** (включая сущность с FK):
   - **Create** — вставка возвращает id (`RETURNING id`), запись появляется в БД.
   - **Read** — список и получение по id; навигационные/lookup-поля по FK заполнены.
   - **Update** — изменение сохраняется; проверить именно сгенерированный `UPDATE` (исторически здесь
     встречались дефекты SQL — напр. пропавшие пробелы вокруг `SET`/`WHERE`).
   - **Delete** — удаление/soft-delete отрабатывает.
3. **Проверить FK-сценарии** — связи (real + virtual FK) резолвятся, нет битых ссылок.
4. **Точечно просмотреть сгенерированный SQL** в `*Repository.cs` на валидность (пробелы, имена колонок,
   quoted-идентификаторы), особенно для таблиц с нестандартными именами.

## Если найден рантайм-дефект шаблона

Документировать для ревью владельца утилиты (контур утилиты), временно обойти на стороне проекта
(контур проекта) — та же политика, что в [`error-classification.md`](error-classification.md).
